# Ojas — Learning Notes

A detailed, stepwise, beginner-level explanation of everything we do while building Ojas — in the exact
order we do it, with the reasoning behind each step. Meant to be read top-to-bottom, or printed for
revision. Every phase of the build gets its own section below.

## Phase 0 — Environment & Project Setup

**Goal of this phase:** get a git repository, a place for backend code, an isolated Python environment,
and a real running database — before writing a single line of application code.

### Step 1 — `git init`: turning the folder into a version-controlled project

**What problem this solves:** as you write code, you will make mistakes, break things, and want to go
back. Without version control, "going back" means manually remembering what you changed, or keeping
copies like `app_v2_final_FINAL.py`. Git solves this properly.

**What Git actually is:** a program that takes **snapshots** of your entire project folder over time.
Each snapshot is called a **commit**. Git stores these snapshots efficiently (it doesn't just copy the
whole folder every time — it tracks changes), and lets you:
- see the full history of every file
- go back to any previous snapshot
- work on experimental changes on a separate "branch" without touching the main code
- (later) push your snapshots to GitHub, so your code is backed up online and shareable

**What the command did:** `git init` was run once, inside `Ojas/`. It created a hidden folder called
`.git/` in that directory. That folder is git's entire database — every commit, branch, and piece of
history lives inside it. As long as `.git/` exists in a folder, git is "watching" that folder (though it
only actually saves a snapshot when you explicitly run `git commit`, which we haven't done yet — we'll
commit once we have real code to save).

**Key mental model:** git only knows about files you tell it to track (`git add`) and changes you
explicitly save (`git commit`). Just having `.git/` there doesn't automatically save anything — think of
it as "the ability to take snapshots exists now," not "a snapshot was taken."

### Step 2 — `.gitignore`: telling git what to never look at

**What problem this solves:** not everything in a project folder should be saved into git history.
Some things are:
- **secrets** (passwords, API keys) — dangerous to store in a history that might get shared/pushed publicly
- **huge, regeneratable folders** — e.g. the Python virtual environment, or (later) `node_modules/` for
  the frontend — these can be perfectly recreated from a small text file (`requirements.txt` /
  `package.json`), so committing the actual (often gigabytes of) files is wasteful and slows git down

**What we wrote**, and why each line is there:
```gitignore
backend/venv/          # the isolated Python environment (Step 4) — regenerate via requirements.txt
backend/__pycache__/   # Python's compiled bytecode cache — auto-regenerated, machine-specific
**/__pycache__/        # same, but matches this pattern anywhere in the repo (not just backend/)
*.pyc                  # individual compiled Python files
.env                   # real secrets for the root-level case
**/.env                # real secrets no matter which subfolder they're in (backend/.env)
frontend/node_modules/ # frontend package installs (Phase 8) — huge, regenerable via package.json
frontend/dist/         # frontend's built/compiled output (Phase 8) — regenerated by a build command
.DS_Store               # a macOS-specific junk file (harmless to ignore even on Windows)
.vscode/                # editor-specific settings, personal preference, not project-relevant
```
**How the syntax works:** each line is a pattern. A trailing `/` means "a folder with this name." A
leading `**/` means "match this at any depth in the folder tree," not just at the root. `*.pyc` uses a
wildcard (`*`) to match any file ending in `.pyc`.

**Important nuance:** `.gitignore` only prevents git from tracking files it **doesn't already know about**.
If a file was already committed, adding it to `.gitignore` later won't remove it from history — you'd
have to explicitly untrack it. Since we wrote `.gitignore` before creating `venv/` or `.env`, we avoided
that problem entirely — always create `.gitignore` first, before generating things you don't want tracked.

### Step 3 — creating the `backend/` folder structure

We ran `mkdir -p backend/app`, creating two nested folders in one command: `backend/` (will hold the
entire Python/FastAPI project) and `backend/app/` inside it (will hold the actual application source
code — routes, models, etc. — starting in Phase 1). Keeping a dedicated `app/` subfolder rather than
dumping Python files directly in `backend/` is a common convention: it separates "project-level" files
(like `requirements.txt`, `.env`, `venv/`) from "importable application code."

### Step 4 — creating the Python virtual environment (`venv`)

**What problem this solves:** Python projects depend on third-party packages (like FastAPI). If you
installed packages "globally" (once, shared by your whole computer), then:
- Project A needing `sqlalchemy==2.0` and Project B needing `sqlalchemy==1.4` would conflict — only one
  version can be installed globally at a time.
- Your computer's Python installation would accumulate every package you ever installed for every
  project, forever.

**The solution — a virtual environment ("venv"):** a self-contained folder containing its own copy of the
Python interpreter and its own separate place to install packages, isolated from both your system Python
and every other project's venv.

**The command:** `python -m venv backend/venv` — this tells Python's built-in `venv` module to create a
new virtual environment at the path `backend/venv`. We verified it worked by listing
`backend/venv/Scripts/` and seeing files including:
- `python.exe` / `pythonw.exe` — this venv's own copy of the Python interpreter
- `pip.exe` — this venv's own copy of the package installer
- `activate` / `Activate.ps1` / `activate.bat` — scripts that, when run, make your terminal use *this*
  venv's `python`/`pip` instead of the system-wide ones (we didn't need to manually "activate" it because
  we called `backend/venv/Scripts/python.exe` and `pip.exe` directly by their full path instead — same
  effect, just more explicit)

**Because `backend/venv/` is in `.gitignore`**, git will never track the (large) contents of this folder.
Anyone who clones this repo instead runs `python -m venv backend/venv` themselves, then installs the
exact same packages from `requirements.txt` (next step) — reproducing an identical environment without
ever transferring the actual files.

### Step 5 — `requirements.txt`: declaring exactly which packages this project needs

This is a plain text file, one package name per line. It is the **recipe** for recreating this project's
Python environment on any machine — instead of shipping the `venv/` folder itself (gitignored, as above),
we ship this tiny file, and `pip install -r requirements.txt` rebuilds the environment from scratch.

Here is what each package we listed actually does, and why it's needed:

| Package | Role | Plain-language explanation |
|---|---|---|
| `fastapi` | Web framework | The library we use to define API endpoints (URLs the frontend will call) as Python functions. Handles turning HTTP requests into Python objects and Python return values back into HTTP responses. |
| `uvicorn` | ASGI server | FastAPI on its own doesn't listen on a network port — it just defines *what to do* when a request arrives. `uvicorn` is the actual process that opens a port (e.g. 8000), accepts incoming connections, and hands each request to FastAPI. ("ASGI" is the standard interface between Python web frameworks and servers — you don't need to memorize this, just know uvicorn is "the thing that actually runs the app.") |
| `sqlalchemy` | ORM (Object-Relational Mapper) | Databases speak SQL (a query language). Instead of writing raw SQL strings everywhere in Python, SQLAlchemy lets us define a Python **class** (e.g. `class JournalEntry`) whose attributes map to database table columns, and then work with normal Python objects — SQLAlchemy translates that into SQL behind the scenes. |
| `alembic` | Migration tool | Works alongside SQLAlchemy. Every time we change our data model (e.g. add a new column), the actual database table needs to be updated to match — a **migration** is a small, version-numbered script describing exactly that change, so the database can be upgraded (or rolled back) in a controlled, repeatable way, instead of manually editing tables by hand. |
| `psycopg2-binary` | PostgreSQL driver | SQLAlchemy doesn't know how to talk to Postgres by itself — it needs a low-level "driver" library that handles the actual network protocol Postgres speaks. This is that driver. (`-binary` means it comes precompiled, so we don't need a C compiler installed to use it.) |
| `pydantic` | Data validation | Used heavily by FastAPI. Lets us declare "this API expects a request body shaped like *this*, with these types" and pydantic automatically checks incoming data matches, rejecting it with a clear error if not — instead of us manually checking every field. |
| `pydantic-settings` | Typed config loading | A small addition on top of pydantic specifically for loading settings (like our `.env` values) into a typed Python object, so misconfigured env vars are caught early instead of causing confusing errors deep in the code. |
| `python-jose` | JWT handling | "JOSE" = JSON Object Signing and Encryption. This library creates and verifies **JWTs** (JSON Web Tokens) — the tokens we'll issue when a user logs in (Phase 2), so the API can verify "this request really is from an already-logged-in user" without looking the user up in the database on every single request. |
| `passlib[bcrypt]` | Password hashing | We must never store a user's actual password in the database — if the database ever leaked, every password would be exposed. Instead we store a **hash**: a one-way scrambled version, using the `bcrypt` algorithm, that can be *checked against* a password but never reversed back into it. |
| `python-dotenv` | Loads `.env` | Reads the `.env` file (Step 6 below) and makes its values available to Python as environment variables at runtime. |

**The install command and what happened:**
```
./venv/Scripts/python.exe -m pip install --upgrade pip -q
./venv/Scripts/pip.exe install -r requirements.txt
```
The first line updates `pip` itself (the installer tool) to its latest version. The second reads
`requirements.txt` top to bottom and, for each package listed:
1. figures out the exact compatible version to use (and every *sub*-dependency that package itself needs
   — e.g. `fastapi` itself depends on `starlette` and `anyio`, which is why the install log showed many
   more packages than we listed — this is called **dependency resolution**)
2. downloads a prebuilt package archive (a "wheel," `.whl` file) for our OS/Python version where available
3. unpacks it into `backend/venv/Lib/site-packages/`

The final line (`Successfully installed ...`) listed *every* package now available inside this venv,
including the ones we didn't list directly but got pulled in automatically.

### Step 6 — `docker-compose.yml`: declaring what infrastructure this project needs

**What Docker actually is, in more depth:** an **image** is a read-only, pre-built package containing an
application plus everything it needs to run (its own tiny OS layer, dependencies, config) — think of it
like an installer/template. A **container** is a running instance of an image — like the difference
between a class and an object in programming, or a recipe vs. an actual cooked meal. You can start
multiple containers from the same image, each isolated from the others and from your actual machine.

**Why we use Docker for Postgres specifically:** installing Postgres directly on Windows means dealing
with Windows-specific installers, service configuration, and version quirks. Running it in a container
means: the exact same `postgres:16` image behaves identically on Windows, Mac, Linux, or a cloud server —
and removing it later is just deleting a container, no leftover system-wide install to clean up.

**Docker Compose** is a tool for describing *multiple* related containers (a whole application's
infrastructure) in one YAML file, and starting/stopping them together with one command, instead of typing
long `docker run` commands by hand for each one. Right now we only have one service (`postgres`), but
later phases will add more (ChromaDB in Phase 5, the FastAPI app itself in Phase 6) into this same file.

**Line-by-line walkthrough of what we wrote:**
```yaml
services:                                  # top-level key: "here is the list of containers to run"
  postgres:                                # our name for this service (we choose this name)
    image: postgres:16                     # use the official "postgres" image, tag "16"
                                            #   (a "tag" pins a specific version, so we don't
                                            #   silently get a different Postgres version later)
    container_name: ojas-postgres          # a friendly, fixed name for the running container
                                            #   (otherwise Docker auto-generates a random name)
    restart: unless-stopped                # if it crashes or the machine reboots, Docker
                                            #   restarts it automatically (unless YOU stopped it)
    environment:                           # environment variables passed INTO the container —
      POSTGRES_USER: ojas                  #   the postgres image's own startup script reads
      POSTGRES_PASSWORD: ojas_dev_password #   these specific variable names and uses them to
      POSTGRES_DB: ojas                    #   create the initial user, password, and database
    ports:
      - "5432:5432"                        # "host_port:container_port" — 5432 is Postgres's
                                            #   standard port. This line makes the container's
                                            #   internal port 5432 reachable from your actual
                                            #   Windows machine at localhost:5432. Without this
                                            #   line, Postgres would run but be completely
                                            #   unreachable from outside its container.
    volumes:
      - ojas_postgres_data:/var/lib/postgresql/data
                                            # Containers are normally "ephemeral" — if you delete
                                            #   a container, everything inside it (including the
                                            #   actual database files) is gone. A "volume" is a
                                            #   piece of storage that lives OUTSIDE the container's
                                            #   lifecycle. This line says "whatever Postgres writes
                                            #   to /var/lib/postgresql/data (its data directory
                                            #   inside the container) should actually be stored in
                                            #   a persistent volume named ojas_postgres_data" — so
                                            #   your data survives even if the container is removed
                                            #   and recreated.

volumes:
  ojas_postgres_data:                      # declares that this named volume exists (Docker
                                            #   manages where it actually lives on disk)
```

### Step 7 — `.env` and `.env.example`: environment-specific configuration

We created two files:
- `backend/.env` — the **real** values this project uses locally (gitignored — never committed)
- `backend/.env.example` — the same variable **names**, with placeholder/fake values, which **is**
  committed to git, purely as documentation of "these are the settings you need to provide yourself"

```
DATABASE_URL=postgresql://ojas:ojas_dev_password@localhost:5432/ojas
SECRET_KEY=dev-only-change-me-before-deploying-a1b2c3d4e5f6
ACCESS_TOKEN_EXPIRE_MINUTES=60
```

- **`DATABASE_URL`** follows a standard connection-string format:
  `postgresql://<username>:<password>@<host>:<port>/<database_name>`. Here that's username `ojas`,
  password `ojas_dev_password` (matching what we told the Postgres container to create in Step 6),
  host `localhost` (our own machine, since Docker exposed the port there), port `5432`, database `ojas`.
  SQLAlchemy will read this single string to know how to connect.
- **`SECRET_KEY`** will be used starting in Phase 2 to cryptographically **sign** JWT login tokens — this
  proves a token was issued by *our* server and hasn't been tampered with. Anyone with this key could
  forge valid login tokens, so it must stay secret and, in a real deployment, be a long random value
  (our placeholder is fine for local development only).
- **`ACCESS_TOKEN_EXPIRE_MINUTES`** will configure how long a login session stays valid before the user
  has to log in again (Phase 2).

**Why this pattern (`.env` + `.env.example`) matters:** it separates "does this code work" (tracked in
git, shared) from "what are today's actual passwords/keys" (never tracked, different per developer/server).
Every environment (your laptop, a teammate's laptop, a production server) can have different real values
in its own `.env`, while the *code* that reads `DATABASE_URL` never changes.

### Step 8 — starting Docker Desktop and bringing up Postgres

Docker requires a background engine ("daemon") to be running before any `docker` command works — having
the Docker Desktop *application* installed isn't enough; it also has to actually be launched and finished
starting up. We checked with `docker ps` (which lists running containers) — it initially failed with a
connection error, meaning the engine wasn't up yet. We started Docker Desktop, waited for it to finish
initializing, and confirmed with `docker ps` again (it then returned successfully, listing containers
from your other projects).

Then: `docker compose up -d postgres` — reads `docker-compose.yml` in the current folder, and starts just
the `postgres` service. The `-d` flag means **detached**: the container runs in the background, and your
terminal is immediately free again (without `-d`, the terminal would stay attached, showing live logs,
until you press Ctrl+C).

The first time this runs for a given image, Docker has to **pull** (download) that image from Docker Hub
(a public registry of images) — you saw a long list of "Pulling fs layer" / "Downloading" / "Download
complete" lines. Images are built from stacked **layers** (each representing one step in how the image
was built, e.g. "install this package"); Docker downloads each layer once and reuses shared layers across
different images, which is why subsequent pulls/rebuilds are much faster. This download only happens
once — the image is now cached locally, so future `docker compose up` calls start instantly.

### Step 9 — verifying it actually works

Two checks, in order of increasing confidence:
1. **`docker ps --filter "name=ojas-postgres"`** — confirms the container exists and its `STATUS` column
   says `Up`, meaning Docker considers the process inside it alive.
2. **`docker exec ojas-postgres psql -U ojas -d ojas -c "SELECT version();"`** — this is the real test.
   `docker exec` runs a command *inside* an already-running container (as opposed to `docker run`, which
   would start a brand-new container). Here we ran `psql` (Postgres's own command-line client, which
   comes bundled inside the postgres image) as user `ojas`, against database `ojas`, executing the SQL
   query `SELECT version();`. Getting back an actual version string
   (`PostgreSQL 16.14 (Debian 16.14-1.pgdg13+1)...`) proves: the container is running, the database
   accepted our exact username/database name from `docker-compose.yml`, and it can execute real SQL —
   i.e., the whole chain (image → container → network port → auth → SQL execution) works end-to-end.

## Phase 1 — FastAPI Skeleton + Database Models

**Goal of this phase:** a running FastAPI application, Python class definitions for every database table
the app needs, and a system (Alembic) for turning those class definitions into real tables in Postgres —
in a repeatable, version-controlled way.

### Concepts first

**What is an API, endpoint, and route?** An **API** (Application Programming Interface) is a set of URLs
a server exposes so *other programs* (our future React frontend, or a test script) can ask it to do
things or fetch data — as opposed to a website meant for a human to click through. An **endpoint** (also
called a **route**) is one specific URL + HTTP method pairing, e.g. `GET /health` or `POST /journal`. Each
one is backed by a plain Python function that FastAPI calls automatically whenever a matching request
arrives.

**SQLAlchemy model vs. Pydantic schema — the difference that trips up most beginners:**
- A **SQLAlchemy model** (Step 4 below) describes a **database table** — its columns, their types, which
  ones are required, foreign key relationships to other tables. This is about *permanent storage*.
- A **Pydantic schema** (Step 5 below) describes the **shape of JSON going in or out of the API** — what
  a request body must contain, what a response will look like. This is about the *API contract*.

They often look similar (both list fields like `name: str`) but serve different jobs, and often
deliberately differ — e.g. a "create journal entry" request shouldn't include an `id` (the database
assigns that automatically), but a "read journal entry" response should.

### Step 1 — folder structure for the FastAPI app

We created three subfolders inside `backend/app/`:
- `app/models/` — will hold one file per SQLAlchemy model (database table definition)
- `app/schemas/` — will hold one file per Pydantic schema (API request/response shapes)
- `app/routers/` — will hold the actual endpoint functions (empty for now — filled in Phase 2/3)

Splitting these into separate folders (instead of one giant file) is a common FastAPI convention: as the
app grows to dozens of tables and endpoints, you can go straight to `app/models/journal.py` instead of
scrolling through one huge file.

### Step 2 — `app/config.py`: typed, validated configuration

```python
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8")

    database_url: str
    secret_key: str
    access_token_expire_minutes: int = 60

settings = Settings()
```
`BaseSettings` (from `pydantic-settings`) is a special Pydantic class that, instead of being filled in
from a JSON request body like normal schemas, automatically reads its values from **environment
variables** — and `env_file=".env"` tells it to also load our `.env` file (Phase 0, Step 7) into those
environment variables first. Declaring `database_url: str` with no default means "this is required — if
it's missing, crash immediately at startup with a clear error" rather than mysteriously failing later
when we first try to use it. `access_token_expire_minutes: int = 60` has a default, so `.env` doesn't
strictly need to set it. The last line, `settings = Settings()`, creates one single instance immediately
when this file is first imported — every other file in the app imports this same `settings` object rather
than re-reading `.env` itself.

### Step 3 — `app/database.py`: the database connection itself

```python
from sqlalchemy import create_engine
from sqlalchemy.orm import declarative_base, sessionmaker
from app.config import settings

engine = create_engine(settings.database_url)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```
- **`engine`** — represents the actual connection *pool* to Postgres (SQLAlchemy doesn't open one
  connection per query; it maintains a small pool of reusable connections for efficiency). Created once,
  reused everywhere.
- **`SessionLocal`** — a **Session** is a single unit-of-work — think "one conversation with the
  database," used to run queries and stage changes. `sessionmaker` is a factory: calling `SessionLocal()`
  produces a new Session bound to our engine. `autocommit=False` / `autoflush=False` mean: nothing is
  actually sent to the database until we explicitly say so — this gives us control over exactly when
  writes happen (important later for things like "only save if every step succeeded").
- **`Base`** — every SQLAlchemy model class (Step 4) will inherit from this. It's how SQLAlchemy collects
  "every table this app knows about" into one place (`Base.metadata`) — which Alembic later reads to
  generate migrations.
- **`get_db()`** — this is a **dependency** (FastAPI-specific pattern): instead of every endpoint function
  manually creating and closing its own database session, it declares "give me a `db` session" as a
  parameter, and FastAPI calls this generator function to produce one. The `yield` (instead of `return`)
  means: give the caller the session, let them use it, and once they're done (even if an error happened),
  run the `finally: db.close()` — guaranteeing every request cleans up its connection. We'll actually wire
  this into endpoints starting Phase 2.

### Step 4 — the SQLAlchemy models (one file per table, in `app/models/`)

Each model is a Python class inheriting from `Base`, where each **class attribute** becomes a **database
column**. We use SQLAlchemy 2.0's modern typed style: `Mapped[int]` says "this column, once loaded, is a
Python `int`," and `mapped_column(...)` configures the actual database-level details (type, nullability,
defaults, foreign keys).

Take `User` (`app/models/user.py`) as the annotated example — every other model follows the same pattern:
```python
class User(Base):
    __tablename__ = "users"                                    # actual table name in Postgres

    id: Mapped[int] = mapped_column(Integer, primary_key=True) # auto-incrementing unique row ID
    email: Mapped[str] = mapped_column(String, unique=True,     # unique=True → Postgres rejects
                                        nullable=False, index=True)  # duplicate emails; index=True
                                                                # makes lookups by email fast
    hashed_password: Mapped[str] = mapped_column(String, nullable=False)
    name: Mapped[str] = mapped_column(String, nullable=False)

    # UI preferences the design spec says must persist per-user (mood, theme, toggles)
    current_mood: Mapped[str] = mapped_column(String, default="calm")
    accent_theme: Mapped[str] = mapped_column(String, default="sky")
    ambient_motion: Mapped[bool] = mapped_column(Boolean, default=True)
    show_mascot: Mapped[bool] = mapped_column(Boolean, default=True)

    cycle_length_days: Mapped[int] = mapped_column(Integer, default=28)
    screen_time_goal_hours: Mapped[float] = mapped_column(Float, default=3.5)

    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), default=lambda: datetime.now(timezone.utc)
    )
```
`default=lambda: datetime.now(timezone.utc)` — a function, not a fixed value — is important: if we wrote
`default=datetime.now(timezone.utc)` directly, that timestamp would be calculated **once**, when the
Python process starts, and every single row would get that same frozen creation time. Using a `lambda`
(an inline anonymous function) means SQLAlchemy calls it fresh, at the moment each row is actually
inserted.

**The other seven tables**, following the exact same pattern, one file each:
| File | Table(s) | Notable columns / relationships |
|---|---|---|
| `journal.py` | `journal_entries` | `content` (the entry text); `sentiment_score`, `mood_label`, `emotion_tags` are nullable — empty until Phase 4's sentiment model fills them in on save |
| `medicine.py` | `medicine_schedules`, `dose_logs` | A schedule has a `time_of_day` + `pill_count` (for refill prediction). Each `DoseLog` row is one specific expected dose, linked to its schedule via `schedule_id`, with `taken`/`taken_at` |
| `habit.py` | `habits`, `habit_entries` | `Habit` is the habit's definition (name/goal/icon); `HabitEntry` is one daily check-in (`kept: bool`, optional `trigger_note`), linked via `habit_id` |
| `screen_time.py` | `screen_time_logs` | One row per day: `date` + `hours` |
| `cycle.py` | `cycle_entries` | One row per logged day: `period_start` flags day-1-of-period, `symptoms` is an array; cycle length lives on `User.cycle_length_days` so day/phase can be *computed* later rather than re-stored everywhere |
| `day_recap.py` | `day_recaps` | Deliberately **no video/audio fields** — only derived numbers (`stress_score`, `energy_score`, `positivity_score`, `confidence`) and text (`transcript_summary`, `overall_read`) are stored, matching the spec's privacy requirement |

**Foreign keys**, e.g. in `journal.py`:
```python
user_id: Mapped[int] = mapped_column(ForeignKey("users.id"), nullable=False, index=True)
```
This tells Postgres: "the value in this column must match an existing `id` in the `users` table" — the
database itself now refuses to save a journal entry pointing at a user that doesn't exist. This is called
a **referential integrity constraint**, enforced by the database, not just by our application code.

**Arrays**, e.g. `emotion_tags: Mapped[list[str] | None] = mapped_column(ARRAY(String), nullable=True)` —
Postgres (unlike most databases) natively supports storing a list of values in a single column. We use
this for tag lists (`emotion_tags`, `symptoms`, `themes`) instead of creating a separate table just to
store a handful of short strings per row.

### Step 5 — `app/models/__init__.py`: why every model must be imported somewhere

```python
from app.models.cycle import CycleEntry
```

### Step 6 — Pydantic schemas (`app/schemas/`): the `Create` / `Read` pattern

Every domain gets (at least) two schemas, e.g. `app/schemas/journal.py`:
```python
class JournalEntryCreate(BaseModel):
    content: str                              # only what the CLIENT must supply

class JournalEntryRead(BaseModel):
    model_config = ConfigDict(from_attributes=True)   # see below

    id: int
    user_id: int
    content: str
    sentiment_score: float | None             # server-computed, may not exist yet
    mood_label: str | None
    emotion_tags: list[str] | None
    created_at: datetime
```
- **`...Create`** schemas define what a client must send when creating something — deliberately minimal
  (no `id`, no `created_at`, no `sentiment_score` — those are the server's job to fill in).
- **`...Read`** schemas define what the API sends back — the full row, including server-generated fields.
- **`ConfigDict(from_attributes=True)`** (this used to be called "orm_mode" in older Pydantic) tells
  Pydantic: "you can build this schema from an object with matching **attributes** (like a SQLAlchemy
  model instance: `journal_entry.content`), not just from a plain dictionary (`journal_entry["content"]`)."
  Without this setting, handing a SQLAlchemy row directly to a Pydantic response schema would fail.

We wrote this same `Create`/`Read` pair for every domain: `user.py`, `journal.py`, `medicine.py` (plus
`DoseLogMarkTaken` for the "mark taken" action), `habit.py`, `screen_time.py`, `cycle.py`, `day_recap.py`.
These aren't wired into any endpoint yet — that's Phase 2/3 — but the shapes are now defined.

### Step 9 — Alembic: setting up migrations

**What a migration actually is:** a small, version-numbered Python script describing one specific change
to the database schema (e.g. "add these 8 tables"), so that schema changes are tracked in git history
just like code changes, can be applied to any environment in the same order, and can even be undone.

**`alembic init alembic`** generated:
- `alembic.ini` — Alembic's own config file (where migration files live, logging format, etc.)
- `alembic/env.py` — the script that actually runs every time you generate or apply a migration; this is
  where we tell Alembic *which* database to connect to and *which* models to compare against
- `alembic/versions/` — the folder where each individual migration script will be saved
- `alembic/script.py.mako` — a template used to generate the boilerplate of each new migration file

## Phase 2 — Auth (JWT / OAuth2)

**Goal of this phase:** real register/login endpoints that issue tokens, and a way to mark any future
endpoint as "logged-in users only."

### Concepts first

**Authentication vs. authorization** — two words that sound similar but mean different things:
- **Authentication** = "who are you?" — proving identity (logging in).
- **Authorization** = "are you allowed to do this?" — e.g. even once logged in, you shouldn't be able to
  read someone *else's* journal entries. Phase 2 only builds authentication; authorization (scoping every
  query to `current_user`) gets applied as we build real endpoints in Phase 3.

**What a JWT actually is.** A **JWT (JSON Web Token)** is a string with 3 dot-separated parts:
`header.payload.signature`. The payload is just JSON (in our case, `{"sub": "priya@example.com", "exp":
...}` — "sub"=subject=whose token this is, "exp"=expiry time) — **not encrypted, just encoded**, so
anyone can read it. What makes it trustworthy is the **signature**: our server computed it using
`SECRET_KEY` (from `.env`) when creating the token, and re-computes it when checking a token — if even one
character of the payload changed, the signature wouldn't match anymore, so tampering is detectable
(though not preventable) without the server ever needing to "remember" a session. This is why `SECRET_KEY`
must never leak: whoever has it can mint valid tokens for any user.

**Why we hash passwords with bcrypt specifically.** Storing plaintext passwords means a single database
leak exposes every user's real password (people reuse passwords across sites, so this is a serious,
common real-world breach pattern). **bcrypt** is a hashing algorithm purpose-built for passwords: it's
deliberately *slow* (computationally expensive) and includes a random "salt" baked into every hash, so
(a) attackers can't brute-force guess passwords quickly even with the hash in hand, and (b) two users with
the identical password get *different-looking* hashes.

### Step 1 — dependency cleanup: `passlib` → `bcrypt` directly, plus two missing packages

We originally listed `passlib[bcrypt]` in `requirements.txt` (Phase 0) as a friendly wrapper around
password hashing. In practice, there's a known version-compatibility issue between recent `bcrypt`
releases and `passlib` 1.7.4. Since `passlib` was just going to call the `bcrypt` library underneath
anyway, we removed `passlib` and call `bcrypt` directly — one less layer, no compatibility risk.

We also discovered two more missing packages **by actually running the app and reading the errors** (a
normal part of backend development — you often only learn a package is needed when something using it
fails to import):
- **`email-validator`** — Pydantic's `EmailStr` type (used in `UserCreate`/`UserRead`, Phase 1) silently
  requires this package to actually validate email format; without it, importing those schemas crashes.
- **`python-multipart`** — FastAPI's `OAuth2PasswordRequestForm` (Step 4 below) parses login credentials
  out of an HTML-form-encoded request body (`username=...&password=...`), which requires this parser
  library to be installed, even though we never write form-parsing code ourselves.

### Step 2 — `app/security.py`: hashing and tokens, in isolation from any web/API code

```python
def hash_password(password: str) -> str:
    return bcrypt.hashpw(password.encode("utf-8"), bcrypt.gensalt()).decode("utf-8")

def verify_password(password: str, hashed_password: str) -> bool:
    return bcrypt.checkpw(password.encode("utf-8"), hashed_password.encode("utf-8"))
```
`bcrypt` works on **bytes**, not Python `str`, hence `.encode("utf-8")` (str → bytes) before hashing and
`.decode("utf-8")` (bytes → str) after, so we can store the result as a normal string column.
`bcrypt.gensalt()` generates a fresh random salt every call — this is *why* hashing the same password
twice gives two different-looking hashes, and it's stored as part of the hash string itself, so
`checkpw` can extract it back out to redo the same computation for comparison.

```python
def create_access_token(subject: str) -> str:
    expire = datetime.now(timezone.utc) + timedelta(minutes=settings.access_token_expire_minutes)
    to_encode = {"sub": subject, "exp": expire}
    return jwt.encode(to_encode, settings.secret_key, algorithm="HS256")

def decode_access_token(token: str) -> str | None:
    try:
        payload = jwt.decode(token, settings.secret_key, algorithms=["HS256"])
    except JWTError:
        return None
    return payload.get("sub")
```
`create_access_token` builds the payload (who this token is for, and when it expires — using our
`ACCESS_TOKEN_EXPIRE_MINUTES` setting from `.env`) and signs it with `HS256` (a standard signing
algorithm) using our secret key. `decode_access_token` does the reverse: verifies the signature is valid
*and* that the token hasn't passed its `exp` time — `jose` raises `JWTError` for either problem, which we
catch and turn into a plain `None` (meaning "not a valid token") rather than letting an exception escape.

Keeping this file free of any FastAPI/database imports is deliberate: hashing and token logic are pure,
self-contained functions — easy to test in isolation, and reusable regardless of how they're eventually
called.

### Step 3 — `app/schemas/auth.py`: the `Token` response shape

```python
class Token(BaseModel):
    access_token: str
    token_type: str = "bearer"
```
`token_type: str = "bearer"` — "Bearer" is the standard way of saying "whoever holds (bears) this token is
authorized" — it's also the exact word we'll see in the `Authorization: Bearer <token>` HTTP header the
frontend will send on every future request.

### Step 4 — `app/dependencies.py`: `get_current_user`, the gatekeeper for protected routes

```python
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="auth/login")

def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)) -> User:
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )
    email = decode_access_token(token)
    if email is None:
        raise credentials_exception
    user = db.query(User).filter(User.email == email).first()
    if user is None:
        raise credentials_exception
    return user
```
This is another FastAPI **dependency** (same pattern as `get_db` from Phase 1). `OAuth2PasswordBearer`
knows how to pull the token out of the incoming `Authorization: Bearer <token>` header automatically —
`tokenUrl="auth/login"` doesn't affect behavior here, it's just metadata so the `/docs` page knows where
to send a login request when you use its "Authorize" button (see Step 7). Any endpoint that declares
`current_user: User = Depends(get_current_user)` as a parameter automatically gets: token extracted →
decoded → matching user fetched from the database — and if *any* of those steps fail, the endpoint
function's body never even runs; FastAPI returns `401 Unauthorized` immediately.

### Step 5 — `app/routers/auth.py`: the three actual endpoints

We introduced **`APIRouter`** here — instead of attaching every route directly to the main `app` object
(Phase 1's `main.py` only had one route, so that was fine), a **router** groups related endpoints into
their own file, which then gets attached to the app in one line. `prefix="/auth"` means every path
declared below is automatically prepended with `/auth` — so `@router.post("/register")` actually serves
`POST /auth/register`.

**Register:**
```python
@router.post("/register", response_model=UserRead, status_code=status.HTTP_201_CREATED)
def register(payload: UserCreate, db: Session = Depends(get_db)):
    existing = db.query(User).filter(User.email == payload.email).first()
    if existing is not None:
        raise HTTPException(status_code=400, detail="Email already registered")
    user = User(email=payload.email, hashed_password=hash_password(payload.password), name=payload.name)
    db.add(user)
    db.commit()
    db.refresh(user)
    return user
```
`payload: UserCreate` — FastAPI automatically parses the incoming JSON body and validates it against our
`UserCreate` schema (Phase 1) before this function even runs; if the JSON is missing `email` or has an
invalid email format, the caller gets a `422` error automatically, and our function body never executes.
We check for an existing user with that email first (can't have two accounts sharing one email). Note we
never store `payload.password` directly — only `hash_password(payload.password)`. `db.add(user)` stages
the new row; `db.commit()` actually writes it to Postgres; `db.refresh(user)` reloads the object from the
database afterward so `user.id` and `user.created_at` (both assigned by the database itself) are
populated on the Python object before we return it. `response_model=UserRead` means FastAPI takes
whatever we `return` (a full SQLAlchemy `User` object, including `hashed_password`!) and filters it down
to *only* the fields listed in `UserRead` before sending the JSON response — this is precisely how the
password hash never leaks out over the API, even though the full object briefly exists in memory.

**Login:**
```python
@router.post("/login", response_model=Token)
def login(form_data: OAuth2PasswordRequestForm = Depends(), db: Session = Depends(get_db)):
    user = db.query(User).filter(User.email == form_data.username).first()
    if user is None or not verify_password(form_data.password, user.hashed_password):
        raise HTTPException(status_code=401, detail="Incorrect email or password", ...)
    access_token = create_access_token(subject=user.email)
    return Token(access_token=access_token)
```
`OAuth2PasswordRequestForm` is a FastAPI-provided shape matching the OAuth2 standard's login form: it has
a `.username` field and a `.password` field (we put the email into `.username` — OAuth2's spec calls it
username, but nothing stops it holding an email address). This is why login uses form-encoded data
(`username=...&password=...`) instead of JSON like every other endpoint — it's specifically to match the
standard so tools like the `/docs` "Authorize" button work automatically. We deliberately return the
*same* error message and status whether the email doesn't exist or the password is wrong — telling an
attacker "that email isn't registered" vs. "wrong password" would leak which emails have accounts.

**Protected test route:**
```python
@router.get("/me", response_model=UserRead)
def read_me(current_user: User = Depends(get_current_user)):
    return current_user
```
This is the simplest possible protected endpoint: it does nothing but require a valid token and return
whoever it belongs to. Every future endpoint that needs to know "which user is making this request" (all
of Phase 3 onward) will use this exact same `current_user: User = Depends(get_current_user)` pattern.

## Phase 3 — Core CRUD Endpoints

**Goal of this phase:** real, working endpoints for the five remaining domains (Journal, Medicine,
Habits, Screen Time, Cycle) — not just saving/reading raw rows, but the *computed* numbers each design
screen actually needs (adherence %, streaks, weekly series, cycle phase), calculated in the backend so
the frontend (Phase 8+) never has to reimplement this logic in JavaScript.

### Concepts first

**REST conventions, briefly.** REST is just a common convention for what HTTP method + URL shape *means*:
`GET` = read (never changes data), `POST` = create something new, `PATCH` = partially update an existing
thing (only the fields you send change — contrast with `PUT`, which conventionally means "replace the
whole thing"). Status codes are the response's one-line verdict: `200` = success, `201` = successfully
*created* something new, `401` = "you're not logged in / invalid token," `404` = "that specific thing
doesn't exist," `422` = "your request body failed validation." We used `201` on every `POST` that creates
a row, and `404` whenever a `PATCH`/lookup targets an ID that doesn't belong to (or exist for) the current
user.

**Why aggregation happens server-side, not in the frontend.** Every screen in the design needs numbers
like "12-day streak" or "87% adherence," not raw rows. We could send the frontend *all* raw rows and make
it compute these in JavaScript, but that means: shipping more data over the network than needed, and
duplicating the same calculation logic in two languages if we ever add a second client (e.g. a mobile
app). Computing it once, in the backend, means every client just displays a number it's handed.

**A pattern used throughout this phase: scope every query to `current_user`.** Every single query below
filters by `.filter(Model.user_id == current_user.id)` (or via a join to something that does). This is
the actual **authorization** step promised back in Phase 2 — without it, a logged-in user could read or
modify *anyone's* data just by guessing IDs.

### Journal (`app/routers/journal.py`)

Three endpoints: `POST /journal` (create), `GET /journal` (list, most recent first, capped with a
`limit` query parameter so a user with thousands of entries doesn't get them all at once), and
`GET /journal/mood-series` — the first real **aggregate** endpoint:
```python
day_col = func.date(JournalEntry.created_at)
rows = (
    db.query(day_col.label("day"), func.avg(JournalEntry.sentiment_score).label("avg_sentiment"))
    .filter(JournalEntry.user_id == current_user.id, JournalEntry.created_at >= since)
    .group_by(day_col)
    .order_by(day_col)
    .all()
)
```
`func.date(...)` truncates a full timestamp down to just its calendar date (so multiple entries on the
same day collapse together). `.group_by(day_col)` is SQL's "bucket rows together by this value" —
combined with `func.avg(...)`, this computes one average sentiment score *per day*, entirely inside
Postgres, in a single query — exactly the shape the Dashboard's mood-trend chart needs. Since Phase 4
hasn't added real sentiment scoring yet, `sentiment_score` is still `null` for every entry, so
`avg_sentiment` currently comes back as `null` too — the query is correct, it just has no real numbers to
average yet. This will start returning real numbers the moment Phase 4 lands, with no changes needed here.

### Medicine (`app/routers/medicine.py`) — the most involved domain

**The "materialize doses on demand" pattern.** A `MedicineSchedule` says "take this at 9 AM, daily" — but
we don't want a background job constantly creating rows for every future day. Instead, `_ensure_doses_for_date()`
is called at the *start* of any endpoint that needs "today's doses to exist," and creates any missing
`DoseLog` row for that exact date, idempotently (checking first, so calling it twice never duplicates):
```python
def _ensure_doses_for_date(db, user, target_date):
    schedules = db.query(MedicineSchedule).filter(..., MedicineSchedule.active.is_(True)).all()
    for schedule in schedules:
        scheduled_for = datetime.combine(target_date, schedule.time_of_day, tzinfo=timezone.utc)
        exists = db.query(DoseLog).filter(DoseLog.schedule_id == schedule.id,
                                            DoseLog.scheduled_for == scheduled_for).first()
        if exists is None:
            db.add(DoseLog(user_id=user.id, schedule_id=schedule.id, scheduled_for=scheduled_for))
    db.commit()
```
`datetime.combine(date, time, tzinfo=...)` glues a calendar date and a time-of-day together into one full
timestamp. `GET /medicine/doses/today` and `GET /medicine/summary` both call this first, so opening either
endpoint on a brand-new day automatically "fills in" that day's expected doses before reading them back.

**Conditional aggregation for the adherence chart** — `GET /medicine/adherence`:
```python
func.sum(case((DoseLog.taken.is_(True), 1), else_=0)).label("taken")
```
`case((condition, value), else_=other_value)` is SQL's if/else *inside* a query. Combined with
`func.sum(...)`, this computes "how many of this day's doses were taken" in the same single grouped query
that also counts the total — this is how the "taken vs. missed" bar chart's data gets built in one
database round-trip instead of one query per day.

**Two lightweight "detected pattern" endpoints, computed with plain Python (not machine learning):**
- `GET /medicine/refill` — a deliberately simple heuristic: `run_out_date = today + pill_count days`
  (assumes one dose per day depletes one pill). We call this a *heuristic*, not a prediction model — it's
  an honest, explainable estimate, upgradeable later if needed.
- `GET /medicine/risk-pattern` — groups every historical *missed* dose by `(day_of_week, hour)` using
  Python's `collections.Counter`, and returns the single most common missed slot. We **verified this
  actually works**, not just that it runs: we inserted three fake missed doses, all on Saturdays at 9 PM,
  directly into Postgres via `psql`, called the endpoint, and got back exactly
  `{"day_of_week":"Saturday","hour":21,"missed_count_at_slot":3,"total_missed":3}` — confirming the
  grouping logic correctly finds a real pattern, not just returning plausible-looking placeholder data.

### Habits (`app/routers/habits.py`) — a real streak algorithm

```python
def _compute_streak(db, habit_id):
    entries_by_date = {e.date: e for e in db.query(HabitEntry).filter(HabitEntry.habit_id == habit_id).all()}
    day = date.today()
    if day not in entries_by_date:
        day -= timedelta(days=1)   # haven't checked in yet today — don't punish the streak for that
    streak = 0
    while day in entries_by_date and entries_by_date[day].kept:
        streak += 1
        day -= timedelta(days=1)
    return streak
```
This walks backward day-by-day from today, counting consecutive `kept=True` entries, and stops the moment
it hits either a day with no entry at all, or a day explicitly marked `kept=False`. **We verified this
with a real gap**: seeded entries for `07-02` (kept), `07-01` (kept), `06-30` (**not** kept), `06-29`
(kept), then checked in for today (`07-03`, kept) — and got back `streak: 3`, correctly stopping at the
`06-30` break instead of continuing on to count the older `06-29` entry too.

`POST /habits/{id}/check-in` is our first **upsert** (update-or-insert): it looks for an existing entry
for that exact date first, and updates it in place if found, rather than creating a duplicate — this
matters because a user might tap "check in" more than once on the same day (e.g. to correct a mistake).

### Screen Time (`app/routers/screen_time.py`)

Simplest domain: log/upsert a hours-value per day (same upsert pattern as Habits), a `/today` endpoint
comparing today's value against `current_user.screen_time_goal_hours` (computing `over_goal: bool`
server-side, so the frontend just checks one boolean instead of doing its own comparison), and a `PATCH
/goal` endpoint for the design's editable goal slider. We tested this with the exact numbers from the
design spec (4.2h logged vs. 3.5h goal) and got `"over_goal": true`, matching the screen's caution-colored
"▲ 0.8h over goal" indicator.

### Cycle (`app/routers/cycle.py`) — a proportional phase model

Rather than storing "which phase is today" directly (which would need updating every single day for every
user), `GET /cycle/status` **computes** it on request from just two things: the most recent logged
period-start date, and the user's `cycle_length_days` setting:
```python
days_since = (date.today() - last_period.date).days
cycle_day = (days_since % cycle_length) + 1
```
The `%` (modulo) handles multi-cycle gaps gracefully — if it's been 40 days since the last logged period
on a 28-day cycle, `40 % 28 = 12`, correctly wrapping into "day 12 of a new cycle" instead of an
impossible "day 40."

For the phase boundaries, we used a small piece of real physiological reasoning: the **luteal phase**
(after ovulation, before the next period) is fairly constant at ~14 days across different total cycle
lengths, while the **follicular phase** (before ovulation) is what actually varies. So instead of fixed
day ranges, we anchor ovulation *relative to the end of the cycle*:
```python
ovulation_center = max(cycle_length - 14, menstrual_end + 1)
```
We verified this against the design's own worked example: logging a period start 14 days before "today,"
with the default 28-day cycle length, correctly returned `cycle_day: 14`, `phase: "ovulation"`, and
`next_period_predicted` 28 days after the logged start date — matching the design screenshot's "day 14 of
28 · Ovulation window" scenario.

### Wiring and full verification

All five routers were added to `main.py` in one line each (`app.include_router(...)`), the same pattern
as Phase 2's `auth.router`. We then ran the whole server once and exercised **every** endpoint above with
real `curl` requests through a real logged-in session (reusing the JWT from Phase 2's test user),
including: creating real rows, reading them back, hitting every aggregate endpoint, deliberately inserting
edge-case data via `psql` to prove the streak/risk-pattern logic handles gaps and patterns correctly (not
just "doesn't crash on the happy path"), and confirming every new route still returns `401` with no token.

## Phase 4 — Sentiment / Mood Scoring (Hugging Face, local)

**Goal of this phase:** every journal entry automatically gets a `sentiment_score`, `mood_label`, and
`emotion_tags` filled in the moment it's saved — no manual tagging by the user, no external API call.

### Concepts first

**What a "pretrained model" means in practice.** Someone else (here, a researcher who published
`j-hartmann/emotion-english-distilroberta-base` on Hugging Face Hub) already spent time and compute
training a neural network on a large labeled dataset of text with emotions attached. We don't train
anything ourselves — we download their already-trained model and just *run* it on our own text
("inference"). This is why we can add a working ML feature with a few lines of code instead of months of
data science work.

**"Local inference" vs. the RAG/Groq layer coming in Phase 5 — the key distinction to hold onto:** this
sentiment model downloads once, then runs **entirely on your own CPU**, with **no internet call per
request and no API key**. Phase 5's chat assistant works completely differently: every single question
sends your text over the network to Groq's servers and needs a paid/free API key. Both are "AI," but one
costs nothing and needs no account (this phase); the other needs an account, a key, and a network round
trip per request (next phase). Recognizing this distinction is important for understanding cost/privacy
trade-offs in real AI products.

### Step 1 — installing PyTorch (CPU-only) and Transformers

`transformers` (Hugging Face's library for using pretrained models) doesn't do the actual number-crunching
itself — it needs an underlying **tensor computation engine**. We chose **PyTorch** (`torch`), the most
common one. The catch: PyPI's default `torch` package bundles NVIDIA GPU (CUDA) support, making it several
gigabytes — massive overkill if you don't have/need a GPU for this. So we installed it explicitly from
PyTorch's own CPU-only package index first:
```
pip install torch --index-url https://download.pytorch.org/whl/cpu
```
This installed `torch-2.12.1+cpu` (note the `+cpu` — that suffix marks it as the CPU-only build) at a
fraction of the size. *Only then* did we run the normal `pip install -r requirements.txt`, which installed
`transformers` itself plus its own dependency chain (`tokenizers`, `huggingface-hub`, `safetensors`,
`numpy`, etc.) — and correctly found `torch` already present, so it didn't try to fetch a different copy.
We documented this two-step order directly as a comment above `torch`/`transformers` in
`requirements.txt`, since it's easy to forget and re-trigger a huge download later.

### Step 2 — `app/sentiment.py`: one model, three derived outputs

**Choosing one multi-label emotion model instead of a single positive/negative one.** A basic
"sentiment-analysis" pipeline only gives POSITIVE/NEGATIVE. We instead used an **emotion classification**
model that scores text against 7 categories at once (`anger`, `disgust`, `fear`, `joy`, `neutral`,
`sadness`, `surprise`) — one model call gives us enough raw signal to derive all three fields our
`JournalEntry` table needs.

**Lazy-loading the model exactly once per running process:**
```python
_emotion_pipeline = None

def _get_pipeline():
    global _emotion_pipeline
    if _emotion_pipeline is None:
        _emotion_pipeline = pipeline("text-classification",
                                      model="j-hartmann/emotion-english-distilroberta-base",
                                      top_k=None)
    return _emotion_pipeline
```
Loading a model into memory is slow (reading a few hundred MB of weights off disk) — we do **not** want
that happening on every single request. This pattern (a module-level variable starting as `None`, filled
in on first use, reused after) means the expensive load happens exactly once, the first time anyone
journals after the server starts, and every request after that reuses the already-loaded model instantly.
`top_k=None` tells the pipeline "give me the score for *every* category," not just the single winning one.

**Turning the model's raw 7 scores into our product's own vocabulary** — this is the part that is
**business logic we wrote**, not something the model gives us for free:
```python
POSITIVE_EMOTIONS = {"joy", "surprise"}
NEGATIVE_EMOTIONS = {"anger", "disgust", "fear", "sadness"}
MOOD_MAP = {"joy": "good", "surprise": "excited", "neutral": "calm",
            "sadness": "sad", "fear": "anxious", "anger": "anxious", "disgust": "anxious"}

def analyze_sentiment(text):
    scores = _get_pipeline()(text)[0]                     # list of {"label", "score"} — all 7
    scores_by_label = {item["label"]: item["score"] for item in scores}

    positive = sum(scores_by_label.get(l, 0.0) for l in POSITIVE_EMOTIONS)
    negative = sum(scores_by_label.get(l, 0.0) for l in NEGATIVE_EMOTIONS)
    sentiment_score = round(positive - negative, 2)        # a single -1..+1-ish number

    top_label = max(scores_by_label, key=scores_by_label.get)
    mood_label = MOOD_MAP.get(top_label, "calm")            # model's 7 labels → our 6 app moods

    ranked = sorted(scores_by_label.items(), key=lambda kv: kv[1], reverse=True)
    emotion_tags = [label for label, score in ranked[:2] if score > 0.15]  # top confident labels only
    return {"sentiment_score": sentiment_score, "mood_label": mood_label, "emotion_tags": emotion_tags}
```
Important lesson here: **the model gives raw signals (7 probabilities); we still had to write the rules**
mapping those onto our app's specific 6 named moods and a single -1..+1 sentiment number. A pretrained
model is a powerful building block, not a finished feature — connecting it to your product's actual
vocabulary is work you always still have to do yourself.

*(Honest caveat, worth knowing: the model's emotion labels — `joy`, `fear`, etc. — are its own fixed
vocabulary, not the exact illustrative tags shown in the design mockup like `#hopeful`/`#restless`. We're
using the closest real signal available rather than inventing fake ones; this is a normal, realistic gap
between a design mockup's placeholder content and what a real model can actually output.)*

### Step 3 — wiring it into `POST /journal`

One-line change to `app/routers/journal.py`:
```python
analysis = analyze_sentiment(payload.content)
entry = JournalEntry(user_id=current_user.id, content=payload.content, **analysis)
```
`**analysis` unpacks the returned dict's three keys directly as keyword arguments matching the
`JournalEntry` model's column names — no other code needed to change, since Phase 1 already defined those
columns as nullable, and Phase 3's `mood_series` aggregate query already worked correctly on whatever
values happened to be there.

## Phase 5 — RAG AI Layer (Groq + LangChain + ChromaDB)

**Goal of this phase:** the actual "Ojas AI" chat — answer a user's question grounded in their own
logged journal/medicine/habit/screen-time/cycle data, citing which entries it used, with a hard
safety fallback for crisis language that never touches the LLM at all.

### Concepts first

**Embeddings, in one sentence:** a model turns a piece of text into a list of numbers (a vector) such
that texts with similar *meaning* end up as nearby vectors — even with completely different words. "I
skipped my walk" and "didn't get any exercise today" would land close together.

**Vector database, in one sentence:** a database built to answer "which of my stored vectors are closest
to *this new* vector?" fast, even across millions of entries — that's how we find a user's most relevant
past entries for a given question, instead of keyword-matching every row.

**RAG (Retrieval-Augmented Generation), the actual mechanism:** rather than asking an LLM to answer purely
from what it memorized during training (risking confident-sounding made-up answers — "hallucination"),
we (1) **retrieve** the user's own relevant logged entries from the vector database, (2) hand them to the
LLM as **context** in the prompt, and (3) instruct the LLM to answer **only** using that context. This is
*why* Ojas can cite real sources instead of inventing plausible-sounding facts.

**Local (Phase 4) vs. hosted (this phase) — the contrast made concrete.** Phase 4's sentiment model runs
on your CPU, no account, no network call per request. This phase's chat needs a real **Groq** account and
API key, and every single chat message is sent over the network to Groq's servers, which run a large
language model (**Llama 3.3**, an open-weight model from Meta) far too big to run on a personal laptop,
and return a generated answer. Groq specifically is known for very fast inference (their own custom chips),
which is why the spec chose it for a responsive chat experience.

### Step 1 — creating a Groq account and API key

You created a free account at console.groq.com and generated an API key (`gsk_...`). Rather than pasting
the raw secret into our conversation, you added it directly to `backend/.env` yourself — the same
gitignored, never-committed file already holding `SECRET_KEY` and the database password.

**A real mistake we caught and fixed along the way:** the key initially ended up in `.env.example`
instead of `.env`. This matters because `.env.example` is the one config file that **is** committed to
git (it's not in `.gitignore` — that's the whole point, it documents which variables exist with safe
placeholder values). A real secret sitting in a file meant to be shared/committed would have been exposed
the moment this repo was ever pushed anywhere. We moved the real key into `.env` and restored
`.env.example` back to a placeholder. Worth remembering as a general habit: **always double-check which
of the two files you're editing before pasting a real secret.**

### Step 2 — ChromaDB as its own Docker service

Added to `docker-compose.yml`:
```yaml
  chromadb:
    image: chromadb/chroma:latest
    container_name: ojas-chroma
    restart: unless-stopped
    ports:
      - "8001:8000"                    # host 8001 → container's own port 8000
    volumes:
      - ojas_chroma_data:/chroma/chroma  # persists vectors across restarts, same idea as Postgres's volume
```
We mapped the **host** side to `8001` (not `8000`) specifically because our own FastAPI app already uses
port 8000 on the host — two different programs can't both listen on the same host port. Inside Docker's
internal network the container still always listens on port 8000; only the "outside door" number changed.
`docker compose up -d chromadb` pulled the image (another one-time download, same pattern as Postgres in
Phase 0) and started it; we verified it was actually alive by hitting its heartbeat endpoint directly:
`curl http://localhost:8001/api/v2/heartbeat` returned a live timestamp.

### Step 3 — installing the Python packages, and an architecture decision worth explaining

`chromadb` (the Python client, to talk to the server we just started), `langchain` (general RAG
orchestration toolkit), and `langchain-groq` (LangChain's specific integration for calling Groq-hosted
models). **A deliberate simplification**: LangChain *can* fully wrap a vector store (embedding + storage +
retrieval) behind its own abstractions, but that path typically pulls in an extra heavy embeddings library
(`sentence-transformers`, another PyTorch-based model). Since we already have `torch`+`transformers` from
Phase 4 and didn't want a second large ML dependency just for embeddings, we instead: use the plain
`chromadb` client directly for storage and retrieval (it ships its own lightweight built-in embedding
model), and use LangChain specifically for what it's genuinely good at here — building the prompt and
calling the LLM. This is still a completely real RAG pipeline (retrieval + augmented generation); we just
didn't route the retrieval half through LangChain's own vector-store wrapper. Real engineering often
involves picking *which* parts of a framework to use rather than adopting 100% of its abstractions.

### Step 4 — `app/vectorstore.py`: storing and searching entries

```python
def _get_collection():
    global _client, _collection
    if _collection is None:
        _client = chromadb.HttpClient(host=settings.chroma_host, port=settings.chroma_port)
        _collection = _client.get_or_create_collection("ojas_entries")
    return _collection
```
Same lazy-singleton pattern as Phase 4's model loading — connect once, reuse the connection. A
**collection** in Chroma is roughly "a table," holding many entries, each with an `id`, the original text
(`document`), and arbitrary `metadata`.

```python
def index_entry(user_id, source, entry_id, entry_date, text):
    collection = _get_collection()
    collection.upsert(
        ids=[f"{source}:{entry_id}"],
        documents=[text],
        metadatas=[{"user_id": user_id, "source": source, "date": entry_date}],
    )
```
`upsert` (update-or-insert, same concept as Phase 3's habit/screen-time/cycle logging) means calling this
twice with the same `id` overwrites rather than duplicates. We never pass a vector ourselves — handing
Chroma the raw `documents` text is enough; **it computes the embedding itself**, using its own default
embedding model, entirely inside this call.

```python
def query_entries(user_id, question, n_results=5):
    collection = _get_collection()
    results = collection.query(query_texts=[question], n_results=n_results, where={"user_id": user_id})
    ...
```
`where={"user_id": user_id}` is the **critical authorization line** for this whole feature — without it,
one user's question could retrieve and leak *another* user's private journal entries just because they
happened to be semantically similar. Chroma filters by this metadata field *before* ranking by similarity.

### Step 5 — `app/crisis.py`: a deliberately "dumb" safety net

```python
CRISIS_KEYWORDS = ["suicide", "kill myself", "end my life", "want to die", ...]

def is_crisis_signal(text):
    return any(keyword in text.lower() for keyword in CRISIS_KEYWORDS)
```
**Why plain keyword matching instead of another ML classifier:** for a safety-critical fallback, being
*predictable and auditable* matters more than being clever. A human can read this exact list and know
precisely what will and won't trigger it; a trained classifier's behavior is much harder to guarantee or
explain. If this fires, we return a fixed, pre-written `CRISIS_MESSAGE` with real helpline numbers —
**the LLM is never even called** in this path, so there's no risk of an AI-generated response to someone
in crisis.

**A real limitation we found while testing, worth keeping in mind:** we first tried the phrase *"I
sometimes think about wanting to die"* — expecting it to match `"want to die"` — and it **didn't**,
because `"want to die"` is not literally a substring of `"wanting to die"` (`"wanting"` ≠ `"want "`).
Simple substring matching is brittle to phrasing and word endings. This is a real, known trade-off of
keyword-based safety nets: predictable, but not exhaustive. A production system would likely combine
this with broader phrase variations and/or a dedicated classifier as an *additional* layer — noted here
as a known gap, not silently papered over.

### Step 6 — `app/rag.py`: the actual retrieval → prompt → LLM chain

```python
SYSTEM_PROMPT = (
    "You are Ojas... Answer ... using ONLY the CONTEXT below... Never provide medical diagnoses "
    "or medical advice — frame any observation as a pattern... If the context doesn't contain "
    "enough to answer, say so honestly instead of guessing.\n\nCONTEXT:\n{context}"
)
prompt = ChatPromptTemplate.from_messages([("system", SYSTEM_PROMPT), ("human", "{question}")])
```
A **prompt template** is just a string with fill-in-the-blank placeholders (`{context}`, `{question}`) —
LangChain substitutes real values in before sending the final text to the LLM. Notice the system prompt
directly encodes two of the spec's Trust & Safety requirements as instructions to the model itself: "only
answer from context" (grounding, reduces hallucination) and "never diagnose, frame as pattern" (matches
the "patterns, not diagnoses" language used throughout the design).

```python
def answer_question(user_id, question):
    if is_crisis_signal(question):
        return {"answer": CRISIS_MESSAGE, "sources": [], "crisis": True}   # LLM never called

    retrieved = query_entries(user_id, question, n_results=5)
    context = "\n".join(f"[{r['source']} · {r['date']}] {r['text']}" for r in retrieved) or "No relevant..."

    chain = prompt | _get_llm()          # LangChain's pipe operator: "run prompt, then feed result to the LLM"
    response = chain.invoke({"context": context, "question": question})
    return {"answer": response.content, "sources": [...], "crisis": False}
```
The `prompt | _get_llm()` syntax is LangChain's **"chain" composition** — `|` (borrowed from Unix pipes)
means "the output of the left side becomes the input of the right side." `chain.invoke({...})` fills in
the template, sends the resulting messages to Groq, and returns the model's response object (`.content`
is the actual generated text). We separately return `sources` (just source name + date, for the
frontend's citation chips), computed straight from what we retrieved — **not** from anything the LLM said,
which keeps the citations trustworthy even if the model's wording is imperfect.

### Step 7 — wiring indexing into every domain's write path, and the chat endpoint

We added one `index_entry(...)` call after each domain's save, turning the row into a short natural-
language sentence first (since Chroma embeds *text*, not raw structured columns):
- **Journal** (`journal.py`): indexes the entry's own content directly.
- **Medicine** (`medicine.py`): on `PATCH /doses/{id}/take`, indexes `"Took {name} ({dosage}) as scheduled."`
- **Habits** (`habits.py`): on check-in, indexes `"Kept habit 'X'"` or `"Slipped on habit 'X' (trigger: ...)"`.
- **Screen Time** (`screen_time.py`): indexes `"Logged {hours} hours of screen time."`
- **Cycle** (`cycle.py`): indexes a sentence built from whichever fields were logged (period start /
  symptoms / notes).

`app/routers/chat.py` is a thin wrapper: one `POST /chat` route, protected by `get_current_user` like
every other endpoint, that just calls `answer_question(current_user.id, payload.question)` and returns it
through the `ChatResponse` schema (`answer`, `sources`, `crisis`).

### Step 8 — verifying it for real, including a subtle bug we hit and fixed

1. **Crisis path, with the fixed phrasing** (`"I just want to die, I can't take this anymore"`) — returned
   the fixed helpline message **instantly**, with `"crisis": true` and empty `sources` — confirming the
   LLM is genuinely never invoked on this path (no network delay at all, unlike every other test below).
2. **The default embedding model's one-time download**: the very first real retrieval call
   (`collection.query(...)`) triggered Chroma's client library to download its own default embedding
   model (`all-MiniLM-L6-v2`, ~79MB) — happening in *our* FastAPI process, not inside the Chroma
   container, since the client computes the query's embedding locally before sending it to the server.
   This is the same "downloaded once into a local cache, reused forever after" pattern we've now seen
   three times (Postgres's image, the Phase 4 sentiment model, and this).
3. **A configuration-reload bug we hit and understood, not just fixed**: right after adding
   `GROQ_API_KEY` to `.env`, a chat request failed with
   `groq.GroqError: The api_key client option must be set...`. The reason: `Settings()` (Phase 1's
   `app/config.py`) only reads `.env` **once, at process startup** — the already-running server process
   had loaded its settings *before* the key existed in the file, and kept that empty value in memory for
   its whole lifetime. Restarting the server made it re-read `.env` from scratch, picking up the key.
   **General lesson**: any time you edit `.env`, the running server needs a restart to see the change —
   it doesn't watch the file live.
4. **Real grounded question**: logged a fresh journal entry ("skipped my evening walk, scrolled my phone,
   felt guilty and tired") and a screen-time log (5.8 hours) through the actual API, then asked
   *"Why have I been feeling tired and guilty lately?"* — Groq's answer directly referenced both the walk
   being skipped and the 5.8-hour screen time figure, and the response's `sources` array correctly listed
   both `Journal` and `ScreenTime` with today's date — a real, end-to-end, cited RAG answer.
5. **An unrelated question** ("What is the capital of France?") — the model correctly refused to answer
   from its own general knowledge and said so explicitly, exactly as instructed by the system prompt —
   confirming the "answer ONLY from context" instruction actually holds in practice, not just in theory.
6. **No token on `/chat`** → `401`, confirming this endpoint is protected like every other.

## Phase 6 — Full Docker Compose + pytest

**Goal of this phase:** (1) package our own FastAPI app into a Docker image so the *entire* backend
(database + vector store + API) starts with a single `docker compose up`, and (2) write an automated test
suite so we can re-verify everything in seconds instead of manually `curl`-ing endpoints.

### Concepts first

**A Dockerfile builds our OWN image.** Until now we've only *used* pre-built images (`postgres:16`,
`chromadb/chroma`). A **Dockerfile** is a recipe for building a new image containing our code plus its
dependencies. Once built, our app is a portable image that runs identically anywhere Docker exists — no
"works on my machine" problems from a missing Python version or package.

**Why automated tests.** We've been testing by hand with `curl` every phase — thorough, but slow and easy
to forget a case. An **automated test** is a function that calls our code and `assert`s the result is what
we expect. `pytest` runs all of them with one command, so after any future change we instantly know if we
broke something. This is also exactly what the CI pipeline in Phase 10 will run automatically on every push.

### Step 1 — `backend/Dockerfile`

```dockerfile
FROM python:3.12-slim            # base image: minimal Debian + Python 3.12
WORKDIR /app                      # all subsequent commands run in /app inside the image

COPY requirements.txt .           # copy ONLY requirements first (see layer-caching note below)
RUN pip install --no-cache-dir torch --index-url https://download.pytorch.org/whl/cpu \
    && pip install --no-cache-dir -r requirements.txt

COPY . .                          # now copy the rest of the source code

CMD ["sh", "-c", "alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000"]
```
- **Why copy `requirements.txt` before the rest of the code:** Docker builds images in **layers**, and
  caches each layer. If we copied everything at once, editing any source file would invalidate the cache
  and force a full re-install of torch/transformers (minutes) on every rebuild. By copying just
  `requirements.txt` and installing *before* copying the code, Docker reuses the cached "dependencies
  installed" layer as long as `requirements.txt` is unchanged — so day-to-day code edits rebuild in
  seconds.
- **`--no-cache-dir`** tells pip not to keep its download cache inside the image (smaller final image).
- **CPU torch, again:** same reasoning as on the host (Phase 4) — the default PyPI torch bundles large GPU
  libraries we don't need; the CPU index URL gives a much smaller build.
- **`--host 0.0.0.0`:** inside a container, `127.0.0.1` would only accept connections from *within* the
  container itself. `0.0.0.0` means "listen on all network interfaces," which is required for the app to
  be reachable from outside the container (via the port mapping in compose).
- **`alembic upgrade head &&` first:** the container applies any pending DB migrations *before* starting
  the server, so a freshly created database is always brought up to the correct schema automatically.

### Step 2 — `backend/.dockerignore`

Just like `.gitignore` but for image builds — lists paths the `COPY . .` step should skip:
`venv/` (huge, and the image installs its own packages anyway), `__pycache__/`, `.env` (we must **not**
bake secrets into an image — the container receives config at *runtime* instead, see Step 3), and
`.pytest_cache/`. Excluding `venv/` especially is important — copying it in would bloat the image and
could even shadow the freshly installed packages with host-specific ones.

### Step 3 — adding the `api` service to `docker-compose.yml`

```yaml
  api:
    build: ./backend                 # build from the Dockerfile in ./backend (vs. a prebuilt image)
    container_name: ojas-api
    restart: unless-stopped
    env_file:
      - ./backend/.env               # load ALL vars from backend/.env (incl. GROQ_API_KEY, SECRET_KEY)
    environment:                      # ...then OVERRIDE just the networking vars:
      DATABASE_URL: postgresql://ojas:ojas_dev_password@postgres:5432/ojas
      CHROMA_HOST: chromadb
      CHROMA_PORT: 8000
    ports:
      - "8000:8000"
    depends_on:
      postgres:
        condition: service_healthy   # wait for postgres's healthcheck to pass first
      chromadb:
        condition: service_started
```
**The single most important idea here — networking differs inside vs. outside Docker.** When we ran the
app on the host, it reached Postgres at `localhost:5432` and Chroma at `localhost:8001`. But *inside*
Docker's private network, containers reach each other by **service name**, on the container's **internal**
port:
- Postgres → `postgres:5432` (not `localhost`)
- Chroma → `chromadb:8000` (the container's real internal port — `8001` was only the *host-side* mapping)

That's why the `api` service loads secrets from `backend/.env` via `env_file`, but then **overrides**
`DATABASE_URL`, `CHROMA_HOST`, and `CHROMA_PORT` in the `environment:` block (which takes precedence over
`env_file`). This avoids duplicating the secret key while still using Docker-internal addresses. Since
`.env` is excluded from the image (Step 2), this is also *how* the container gets its config at all —
injected at runtime, never baked in.

**`depends_on` with `condition: service_healthy`.** Plain `depends_on` only waits for a container to
*start*, not to be *ready* — Postgres's process starts a moment before it actually accepts connections, so
our `alembic upgrade head` could fire too early and crash. To fix that we added a **healthcheck** to the
postgres service:
```yaml
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ojas -d ojas"]   # postgres's own "am I ready?" tool
      interval: 5s
      timeout: 5s
      retries: 5
```
and made `api` wait for `condition: service_healthy`, so migrations only run once the DB genuinely accepts
connections. (Chroma only needs `service_started` because the app doesn't touch it until the first
chat/index request, well after startup.)

### Step 4 — the test suite (`backend/tests/`)

We created a separate test database `ojas_test` in the same Postgres container
(`CREATE DATABASE ojas_test;`) so tests never touch real dev data. We used a real Postgres test DB rather
than the usual quick in-memory SQLite because our models use Postgres-specific `ARRAY` columns that SQLite
can't reproduce — the test environment must match production's database engine here.

**`tests/conftest.py`** — shared pytest **fixtures** (reusable setup). The critical ordering detail:
```python
os.environ["DATABASE_URL"] = "postgresql://ojas:ojas_dev_password@localhost:5432/ojas_test"
os.environ["SECRET_KEY"] = "test-secret-key"
os.environ["GROQ_API_KEY"] = "test-groq-key"
# ...only AFTER setting these do we import app modules:
from app.database import Base, get_db
from app.main import app
```
Because `app.config` reads settings *once at import time*, we must set the test env vars **before** any
app module is imported. Environment variables take precedence over the `.env` file in pydantic-settings
(we verified this empirically before running anything destructive, since the fixtures drop tables — we did
*not* want that pointed at the real `ojas` database by accident). Key fixtures:
- `fresh_database` (marked `autouse=True`, so it runs for *every* test automatically): drops and recreates
  all tables before each test, guaranteeing tests are isolated and never depend on each other's leftover data.
- `app.dependency_overrides[get_db] = _override_get_db`: this is the **payoff** of having made `get_db` a
  dependency back in Phase 1 — FastAPI lets us swap it out in tests so every endpoint transparently uses
  the *test* database session instead of the real one, without changing any endpoint code.
- `client`: a `TestClient` that calls the app in-process (no real network/server needed).
- `auth_headers`: registers + logs in a test user and returns the `Authorization` header, so protected
  endpoints can be tested easily.

**`tests/test_auth.py`** — register returns a user with no password field leaked, duplicate email → 400,
login + access `/auth/me`, protected route → 401 without a token, wrong password → 401.

**`tests/test_journal.py`** — demonstrates **mocking** (`monkeypatch`): we replace `analyze_sentiment` and
`index_entry` *as imported into `app.routers.journal`* with instant fakes, so these tests don't load the
heavy HF model or require the Chroma container. Important subtlety: you patch a name **where it's used**
(`app.routers.journal.analyze_sentiment`), not where it's defined (`app.sentiment.analyze_sentiment`),
because `from app.sentiment import analyze_sentiment` created a *new binding* in the journal module.

**`tests/test_chat.py`** — the crisis-safe path needs **no** mocking (the guard returns before any
retrieval/LLM call, so we assert `crisis=True`, empty sources, and a helpline message directly). The
grounded path mocks *both* `app.rag.query_entries` (returns two known fake sources) and `app.rag._get_llm`
(returns a tiny fake `RunnableLambda` producing a fixed `AIMessage`) — this lets us verify the full wiring,
especially that the response's `sources` come out correctly, **without** calling Groq over the network.

### Step 5 — verification

- `pytest -v` → **11 passed** in ~12 seconds (2 harmless deprecation warnings). Every test hit the
  isolated `ojas_test` database, and none required Groq, the HF model, or the Chroma container to be up.
- `docker compose build api` → built our application image (installing torch/transformers/chromadb inside
  it — a large, one-time build). Then `docker compose up` brings up all three services together
  (postgres → healthcheck passes → api runs migrations → api serves; chromadb alongside), giving a genuine
  one-command startup of the entire backend from nothing.

## Phase 7 — External Integrations (YouTube · AWS S3 · Google Calendar)

**Goal of this phase:** wire in three real third-party services — mood-based video recommendations
(YouTube), PDF data export (AWS S3), and medicine-schedule sync (Google Calendar) — each built so it works
*immediately* with a safe stub and flips to the real service the moment credentials are provided.

### Concepts first — three different authentication models

The single most useful lesson of this phase: different external services authenticate differently
depending on *whose* data they touch.
- **API key (YouTube)** — one secret string attached to each request. Used when the data is public (any
  YouTube video), not tied to a specific person's private account. Simplest.
- **App credentials (AWS S3)** — an access-key-ID + secret-access-key pair identifying *your application*
  to AWS, so it can read/write files in *your* storage bucket. Medium.
- **OAuth 2.0 (Google Calendar)** — because it touches a *user's own private calendar*, the user must
  explicitly grant consent through Google's own screen; your app then receives a token proving "this
  specific user allowed me to act for them." Most complex, and the same flow behind every "Log in with
  Google" button. We'll cover the mechanics when you set up the account.

**The stubbing strategy (why we can build now without accounts).** Every integration checks whether its
credentials are configured; if not, it returns realistic fake/local behavior instead of failing. This is
the same pattern we discussed from the start — it lets the whole app be built and tested end-to-end now,
and each service is switched to "real" independently, later, just by dropping a key into `.env`.

### Step 1 — dependencies and config

Added `boto3` (AWS SDK), `fpdf2` (PDF generation), and the Google libraries
(`google-auth`, `google-auth-oauthlib`, `google-api-python-client`). Extended `app/config.py` with
optional settings (all defaulting to empty strings so nothing is *required*): `youtube_api_key`, the four
`aws_*` / `s3_bucket` values, and the three `google_*` OAuth values. Empty = "use the stub."

### Step 2 — YouTube (`app/integrations/youtube.py` + `app/routers/discover.py`)

A `MOOD_QUERIES` dict maps each of the app's six moods to a wellness search phrase (e.g. `anxious` →
`"anxiety relief calming breathing"`). `get_recommendations(mood)`:
- **No key configured** → returns 6 deterministic stub cards (`is_stub: True`), so the Discover screen
  renders immediately.
- **Key configured** → calls the real YouTube Data API `search` endpoint over `httpx`, extracting each
  video's id, title, channel, and thumbnail URL.
`GET /discover` defaults to the logged-in user's own `current_mood` if no `?mood=` is passed. We verified
the stub path returns 6 cards with the correct mood-mapped titles.

### Step 3 — AWS S3 + PDF export (`app/integrations/storage.py` + `app/routers/export.py`)

`generate_pdf(user, db)` queries the user's real logged data (journal count + latest mood, dose totals +
adherence %, habit count) and lays it out into a one-page PDF with `fpdf2`, including a footer explicitly
stating it contains no diagnoses (the Trust & Safety requirement, carried through even to the export).
`upload_to_s3(data, key)`:
- **No S3 configured** → returns `None`.
- **S3 configured** → uploads the bytes with `boto3` and returns a **presigned URL** (a temporary,
  time-limited download link — expires in 1 hour — so files aren't left publicly readable).
`POST /export` calls both: if it got a URL back it returns `{"url": ...}`; otherwise it returns the PDF
**file directly** as a download (`Content-Disposition: attachment`). So export fully works with no AWS
account at all — you just get the file inline instead of a link.

**A real bug we hit and fixed (twice):** `fpdf2`'s default Helvetica font only supports the latin-1
character set, and we'd used a Unicode **em-dash** (`—`) in both a stub video title and the PDF heading.
The PDF export crashed with `FPDFUnicodeEncodingException` on that character. Fix: use a plain hyphen
(`-`). General lesson — the default PDF fonts are *not* full-Unicode; either stick to basic characters or
register a Unicode TTF font. We saw the same character break both a string headed for a PDF and one only
displayed in the terminal, a good reminder that "it's just a dash" can genuinely break things downstream.

### Step 4 — Google Calendar OAuth (`app/integrations/google_calendar.py` + `app/routers/integrations.py`)

This needed **a schema change first**: storing per-user Google tokens. We added three nullable columns to
the `User` model (`google_access_token`, `google_refresh_token`, `google_token_expiry`) and generated +
applied a **second Alembic migration** (`f0b39cecc56c`) — our first schema change since the initial
`create core tables`, and a good demonstration that autogenerate correctly detected *only* the three
added columns and nothing else.

The module (all guarded by `is_configured()`, which returns False when Google credentials are absent):
- `get_authorization_url()` builds the Google consent-screen URL via `google-auth-oauthlib`'s `Flow`,
  requesting `access_type=offline` + `prompt=consent` so Google returns a long-lived **refresh token**
  (access tokens expire hourly; the refresh token lets us get new ones without re-prompting the user).
- `exchange_code_and_store(code, user, db)` trades the one-time `code` Google sends back for real tokens
  and saves them on the user.
- `sync_medicine_schedules(user, db)` builds a Calendar client from the stored tokens and inserts a daily
  recurring event per active medicine schedule.
The router exposes `/integrations/google/authorize`, `/callback`, and `/sync` — all returning `503 Not
Configured` cleanly when credentials are absent (verified). The `/callback` is intentionally left as a
documented stub (`501`) since fully wiring it needs the OAuth `state` round-trip to tie the redirect back
to the logged-in user — to be completed once you set up the Google Cloud project.

### Step 5 — tests

Added `tests/test_integrations.py`: Discover returns 6 stub videos, Export returns a real PDF (asserts the
`%PDF-` magic bytes) when S3 isn't configured, Google authorize returns `503`, and Discover requires auth.
Full suite now **15 passing**.

### Step 6 — a genuine Docker debugging saga (and how we diagnosed it)

After adding the Phase 7 code we rebuilt the `api` image and restarted the container — but its Phase 7
routes didn't work. This turned into a multi-layer debugging exercise worth recording in full, because the
process matters more than any single command:

1. **Symptom:** Phase 7 endpoints failed against the container (but worked via host uvicorn, and all 15
   tests passed). So the *code* was fine — something about the *container* was stale.
2. **Confirming the image was stale:** we compared files **on disk** vs. **inside the image** directly:
   `docker run --rm --entrypoint sh ojas-api:latest -c "ls app/routers/"` — the image was missing
   `discover.py`, `export.py`, `integrations.py`, and the new migration. `docker images` showed the image's
   creation time hadn't changed. So our "rebuilds" weren't producing a new image at all.
3. **A crash-loop clue:** the container also kept restarting because its startup `alembic upgrade head`
   reported *"Can't locate revision `f0b39cecc56c`"* — the **database** had been migrated to that revision
   (applied from the host), but the stale **image** only contained the older migration. A container whose
   code lags behind the DB's applied migration version will refuse to start.
4. **The real root cause:** the Docker **build itself was failing** — `failed to read dockerfile: invalid
   file request Dockerfile` — a BuildKit error. We only *saw* this once we read the raw build output from
   the start; earlier we'd piped build output through `| tail`, and **the pipe's exit code (tail's `0`)
   masked the build's real failure**, so the build looked like it "succeeded" while doing nothing, and
   `docker compose up` just re-ran the old existing image.
5. **The fix:** this BuildKit error is commonly triggered by the project living under a OneDrive-synced
   path (`C:\Users\USER\OneDrive\...`), whose virtual filesystem interferes with BuildKit's context read.
   Building with the **legacy builder** — `DOCKER_BUILDKIT=0 docker build -t ojas-api:latest .` — bypasses
   BuildKit entirely and reads the context correctly.

**Lasting lessons from this one:** (a) never pipe a build/critical command through `| tail` if you need its
exit code — the pipe reports the *last* command's status, hiding failures; check the raw output or
`PIPESTATUS`. (b) When a rebuild "succeeds" but behaves like old code, verify the image's *actual contents*
rather than trusting the exit message. (c) App code (migration files) and the database's recorded migration
version must stay in lockstep. (d) Tooling on Windows under OneDrive paths can behave differently — moving
the project outside OneDrive, or using the legacy builder, avoids a class of these issues.

We also saw the api container get **OOM-killed** (`Exited (137)`, i.e. SIGKILL from out-of-memory) when run
alongside Postgres + Chroma — the ML libraries (torch/transformers) are memory-hungry. For day-to-day
development the lighter, faster path is to run the backend via the host `uvicorn` (in the venv) while only
Postgres and Chroma run in Docker; the full containerized stack is mainly for the one-command demo and for
deployment (Phase 10). If the container is needed, raise Docker Desktop's memory limit in its settings.

## Phase 8 — Frontend Setup + App Shell

**Goal of this phase:** stand up the React frontend (a brand-new codebase in TypeScript, not Python) and
build the persistent "shell" every screen lives inside — sidebar, top bar, the glowing mascot, and the
mood-tinting system — matching the design tokens exactly.

### Concepts first (this is a whole new stack)

- **React** — a JavaScript library for building UIs from reusable **components**. A component is just a
  function that returns markup; you compose small ones (a button) into big ones (a screen). When data
  changes, React re-renders only what changed.
- **TypeScript** — JavaScript + **types** (like Python's type hints, but checked at build time). Catches
  "you passed a number where a string was expected" before the code runs.
- **JSX** — the HTML-looking syntax inside components (`<div className="…">`). It compiles to JavaScript;
  note it's `className`, not `class` (because `class` is a reserved word in JS).
- **Vite** — the dev server + build tool (the frontend's `uvicorn`). Instant hot-reload while developing,
  optimized bundle for production.
- **Tailwind CSS** — style with small utility classes in the markup (`className="flex gap-4 rounded-xl"`)
  instead of separate `.css` files. We map the design's exact tokens into Tailwind's theme.
- **npm** — Node's package manager (the JS `pip`); packages live in `node_modules/` (gitignored).
- **Hooks** — special React functions starting with `use`: `useState` (component-local state),
  `useEffect` (run code when something changes, e.g. sync to the DOM/localStorage), `useContext` (read
  shared app-wide state). We use all three.

### Step 1 — scaffolding

`npm create vite@latest frontend -- --template react-ts` generated a React+TypeScript project under
`frontend/`. Then we added the libraries the app needs:
- **react-router-dom** — client-side routing (switch screens without a full page reload).
- **@tanstack/react-query** — fetch + cache API data with automatic loading/refetch handling (used
  heavily in Phase 9).
- **axios** — HTTP client for calling our backend.
- **tailwindcss + @tailwindcss/vite** — styling (Tailwind v4, wired in as a Vite plugin).
- **lucide-react** — the line-icon set the design recommends.

### Step 2 — design tokens → `src/index.css`

This is where the design handoff's exact values live. Two layers:
- **`@theme { … }`** (Tailwind v4's config-in-CSS) defines the stable palette and fonts as utility
  classes: `text-ink` (#22314F), `bg-surface`, `bg-soft`, `text-caution`, `font-display` (Spectral), etc.
- **`:root { … }`** defines the *runtime-changeable* variables — `--accent`/`--accent-deep`/`--accent-soft`
  and `--mood-tint`/`--mood-glow` plus mascot animation speeds. The accent utilities in `@theme`
  *reference* these (`--color-accent: var(--accent)`), so when JavaScript changes `--accent` on the fly,
  every `bg-accent`/`text-accent-deep` element updates live via the CSS cascade — no re-render needed.
- We loaded the three Google Fonts (Spectral, Figtree, JetBrains Mono) and defined the design's
  keyframes (`breathe`, `floaty`, `blink`, `glowpulse`, `rise`, `typing`), plus a global
  `prefers-reduced-motion` rule that disables all animation for users who ask their OS for less motion.

### Step 3 — the theme data (`src/data/theme.ts`) and nav (`src/data/nav.ts`)

Plain typed data, separated from components so it's reused in multiple places:
- `MOODS` — the six moods with emoji, tint, glow, and float/breathe durations (straight from the spec's
  table); `ACCENTS` — the three accent themes (Sky/Periwinkle/Aqua) with their three colours each.
- `NAV_ITEMS` — one entry per screen (path, label, top-bar title, icon, optional NEW badge). This single
  list drives **both** the sidebar links **and** the router routes, so they can never drift out of sync.

### Step 4 — app-wide state (`src/context/AppContext.tsx`)

A React **Context** holds `mood`, `accent`, `ambientMotion`, `showMascot` — the settings the spec says
must persist per user. Any component can read/update them via a `useApp()` hook. Two important patterns:
- **Initialising from localStorage** (`useState(() => persisted('ojas_mood', 'calm'))`) so a returning
  user keeps their settings.
- **`useEffect` side-effects**: whenever `mood` changes, an effect writes the mood's tint/glow/speeds onto
  the `<html>` element's CSS variables *and* saves to localStorage. This is the bridge between React state
  and the CSS-variable theming from Step 2 — change the mood, and the whole app re-tints with a 0.9s
  transition.

### Step 5 — the shell components

- **`Mascot.tsx`** — the glowing orb, pure CSS/SVG (radial-gradient body + aura + per-mood eye/mouth SVG
  paths), animated with the mood's `breathe`/`floaty`/`blink`/`glowpulse` speeds. Respects the
  `showMascot` and `ambientMotion` toggles (the latter maps to `animationPlayState: paused`).
- **`Sidebar.tsx`** — 248px fixed column: the orb-as-"O" logo lockup + "jas" + "Your companion" tagline,

## Phase 9 — Frontend Screens Wired to the Real API

**Goal of this phase:** replace the 9 placeholder routes with real screens, each fetching live data from
the backend, plus a proper login/sign-up page. This is where the app becomes genuinely usable.

### Concepts first

- **React Query (`@tanstack/react-query`)** — the data layer. Instead of manually tracking loading/error
  state and re-fetching, we write a small hook per endpoint. `useQuery` = read data (cached, auto-refetch);
  `useMutation` = change data (POST/PATCH), then `invalidateQueries` tells the cache "this data changed,
  re-fetch it" so the UI updates itself. Every screen uses this.
- **The custom-hook pattern (`src/lib/hooks.ts`)** — one hook per endpoint (e.g. `useMedicineSummary`,
  `useDosesToday`, `useMarkDoseTaken`). Screens just call the hook and read `.data`/`.isPending`. Keeps
  components clean and API calls reusable/typed.
- **Recharts** — the charting library. We compose a chart from pieces (`<AreaChart><Area/><XAxis/></AreaChart>`)
  wrapped in `<ResponsiveContainer>`. We set `isAnimationActive={false}` on the data series (avoids a
  render-loop that stalls headless screenshots, and it's snappier).

### Step 1 — Login / Auth (the foundation everything needs)

The screens can't fetch anything without a logged-in user, so we built auth first:
- `src/lib/api.ts` already attaches the JWT from localStorage on every request (Phase 8). We added
  `src/screens/Login.tsx` (calls `POST /auth/register` and/or `POST /auth/login`, stores the token,
  redirects) and `src/components/RequireAuth.tsx` — a route guard that bounces to `/login` if there's no
  token. `App.tsx` wraps all app routes in `<RequireAuth>`.
- **Design update mid-phase:** a dedicated login/sign-up design was provided (`AUTH_LOGIN_SIGNUP_SPEC.md`).
  We rebuilt `Login.tsx` to match it exactly — one component with `mode: 'signin' | 'signup'`, a fixed
  sky-blue orb (`AuthMascot.tsx`), "Continue with Google" button, an "or" divider, styled fields
  (Full-name only in signup, "Forgot?" only in signin), the primary submit, a legal line (signup only),
  and a swap line to toggle modes. Verified both modes against their reference screenshots.

### Step 2 — a beginner gotcha we hit repeatedly: controlled inputs

React **controlled inputs** (`<input value={x} onChange=...>`) only update their state from a real React
input event. When testing by setting `input.value` directly in the browser, React's state stayed empty and
the form submitted blank. The fix (for testing) is to use the native value setter + dispatch an `input`
event so React notices. Real users typing are unaffected — this only bit our automated checks. Good to know
the mechanism, though: the DOM value and React's state are two separate things that must be kept in sync via
events.

## Phase 9.5 — Making the App Responsive

**Goal:** the design was desktop-only (fixed sidebar + multi-column grids), so on a phone the sidebar ate
the screen and content was crushed. We added mobile support without touching the desktop look.

### The one core concept: Tailwind is mobile-first

Base utility classes apply to **every** screen size. Prefixes (`sm:` ≥640px, `md:` ≥768px, `lg:` ≥1024px)
**override** them only at that width and up. So you write the *phone* layout as the default, then add
`lg:` classes for the wide layout. Examples we used:
- `grid-cols-2 lg:grid-cols-4` → 2 columns on phones, 4 on large screens (Dashboard stat cards).
- `grid-cols-1 lg:grid-cols-[1.5fr_1fr]` → stacked on phones, a 1.5-to-1 two-column split on desktop.
  (Arbitrary values use `_` for the space: `[1.5fr_1fr]`.)
- `flex-col sm:flex-row` → stack vertically on phones, side-by-side on wider screens (hero card).
- `hidden sm:inline` / `sm:block` → hide something on the smallest screens, show it when there's room
  (the date sub-line, the mood label text).

### The sidebar → mobile drawer

The biggest change. The sidebar was a fixed 248px column. We made it:
- **Desktop (`md:` and up):** `md:static md:translate-x-0` — a normal in-flow column, always visible.
- **Mobile:** `fixed inset-y-0 left-0 z-40 -translate-x-full` — pulled off-screen to the left by default,
  and slid in (`translate-x-0`) with a `transition-transform` when open. A dark backdrop
  (`bg-black/30 md:hidden`) sits behind it and closes it on tap; tapping a nav link closes it too.
- The state (`navOpen`) lives in `AppShell`, toggled by a new **hamburger button** in the top bar that's
  only visible on mobile (`md:hidden`). Tailwind's responsive classes do the showing/hiding — no JS width
  checks needed.

### Every multi-column grid got responsive

We converted each screen's grid: 4-col stat cards → `grid-cols-2 lg:grid-cols-4`; 3-col card rows →
`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`; the video grid → `grid-cols-2 lg:grid-cols-3`; and all the
two-panel screens (Journal, Medicine, Chat, Cycle, Screen Time, Day Recap) from a fixed `1.5fr 1fr`
inline style → a stacking `grid-cols-1 lg:grid-cols-[…]` class. Charts already used `ResponsiveContainer`,
so they adapted for free once their container width changed.

### A verification lesson: know your test viewport

The preview pane's "desktop" mode was only **577px** wide — below the `md`/`lg` breakpoints — so it kept
*showing the mobile layout*, which briefly looked like the desktop layout was broken. It wasn't: forcing a
real **1280px** viewport confirmed the desktop layout (two columns, static sidebar, no hamburger) was
intact, while 375px showed the mobile layout (drawer, stacked cards, 2×2 stats). Lesson: a responsive bug
and "you're just testing at the wrong width" look identical — always check `window.innerWidth` before
concluding something's broken.

### Verified
`npm run build` clean; at **375px**: hamburger + slide-in drawer + backdrop, hero stacks, stats 2×2, panels
stack. At **1280px**: full sidebar, two-column splits side-by-side, no hamburger. Same components, driven
entirely by Tailwind breakpoint prefixes.

## Phase 10 — CI/CD + Deployment

**Goal:** automated checks on every push (CI), and get the app online (frontend + backend).

### Concepts first

- **CI (Continuous Integration)** — every time you push code, a server automatically runs your tests +
  build. You find out immediately if something broke, instead of discovering it later. **CD (Continuous
  Deployment)** extends that to auto-deploying when code lands.
- **GitHub Actions** — GitHub's built-in CI. You describe jobs/steps in a YAML file under
  `.github/workflows/`; GitHub spins up fresh Linux VMs and runs them on each push. Free for public repos.

### Step 1 — the CI workflow (`.github/workflows/ci.yml`)

Two parallel jobs:
- **backend** — spins up a throwaway **Postgres service container** (matching what `conftest.py` expects:
  user `ojas`, db `ojas_test`), installs deps (CPU torch + `requirements.txt`), and runs `pytest`. A
  `services:` block with a health-check ensures Postgres is ready before tests run.
- **frontend** — `npm ci` (a clean, lockfile-exact install) then `npm run build` (which type-checks *and*
  bundles, so a type error fails CI).
Both use built-in caching (`cache: pip` / `cache: npm`) so repeat runs are faster.

### Step 2 — making the app production-ready (dev vs. prod differ)

Locally, the frontend and backend share an origin via the Vite dev proxy. In production they're on
**different domains** (Vercel vs. Railway), which changes two things:
- **CORS (Cross-Origin Resource Sharing)** — browsers block a page on domain A from calling an API on
  domain B unless the API explicitly allows it. We added FastAPI's `CORSMiddleware`, driven by a new
  `CORS_ORIGINS` setting (comma-separated allowed origins). Locally that's `localhost:5173`; in prod you
  add the Vercel URL.
- **API base URL** — the frontend's axios client now uses `import.meta.env.VITE_API_URL ?? '/api'`: it
  falls back to the dev proxy locally, but in production reads `VITE_API_URL` (the deployed backend URL,
  set in Vercel). `VITE_`-prefixed vars are the only ones Vite exposes to browser code.
- **SPA routing** (`frontend/vercel.json`) — a rewrite sending every path to `index.html`, so refreshing
  on a deep link like `/dashboard` doesn't 404 (React Router handles the path client-side).

### Step 3 — a test-isolation bug the prod work surfaced

Adding real integration keys to `.env` earlier made two tests fail locally: `test_discover...` expected
*stub* videos but got real ones, and `test_google_authorize...` expected a 503 but Google was now
configured. The tests were accidentally reading the developer's real `.env`. Fix: `conftest.py` now forces
`YOUTUBE_API_KEY`, `GOOGLE_*`, `AWS_*` to empty during tests, so they're **deterministic** regardless of
local secrets (and in CI there's no `.env` anyway). Lesson: tests must not depend on developer-specific
config — pin every relevant setting in the test fixture.

### Step 4 — the deployment plan (`DEPLOYMENT.md`)

- **GitHub** first (everything deploys from it; CI runs there).
- **Frontend → Vercel**: import the repo, root `frontend`, set `VITE_API_URL`, deploy. Easy + free.
- **Backend → Railway** (not Render's 512 MB free tier): the image is ~2 GB because of PyTorch + Hugging
  Face, so it needs real RAM. Railway builds from our `Dockerfile`, plus a managed Postgres and a ChromaDB
  service; env vars wire them together; `alembic upgrade head` runs on container start. An honest
  future optimization: swap the local PyTorch sentiment model for a hosted API to shrink the image enough
  for smaller/free tiers.

### Verified
`pytest` → 15 passing (after the conftest fix); `npm run build` clean; backend imports fine with the CORS
middleware. The CI workflow, prod config, and deploy guide are all in place — the remaining steps
(GitHub push + Vercel/Railway account setup) are done through those platforms' dashboards.
